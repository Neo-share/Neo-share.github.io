当 AI 读不懂你的 PRD：用领域 DSL 架起产品语言与代码之间的桥

当 AI 读不懂你的 PRD：用领域 DSL 架起产品语言与代码之间的桥

1. 先讲结论

如果你把 PRD 丢给 Cursor，让它「按需求改订单模块」，它大概率会做对一半、漏掉一半。

不是模型不够聪明，而是产品语言和代码实现之间，本来就隔着一层翻译损耗。AI 没参加过你们的需求会，默认只能读代码和零散文档——很容易把「订单取消」理解成某个 cancelOrder 函数，漏掉退款、库存、通知；或者把 PRD 里的规划当成已实现，改完才发现动错了层。

我的结论是：别让 AI 靠猜。 在 PRD 和代码之间，用一组小 DSL（术语、状态机、规则、验收）加上差距文档，把「用户说的」和「代码里的」钉在同一张网上。对 AI 而言，元数据（status/gap）+ 映射（code/tests）+ 治理（冲突听谁的），和规则语义本身一样重要。

2. 问题的本质：一句话，六条路径

做过订单系统的人，大概都见过这种 PRD：

用户可在支付后 30 分钟内取消未发货订单，全额退款并收到通知。

落到代码里，至少拆成六个问题：

维度 隐含问题
概念 「取消」和「撤销订单」是不是一回事？
状态 pending 和 paid 取消逻辑是否相同？
规则 已发货、超 30 分钟怎么处理？
权限 买家能点取消吗？管理员能否强制取消？
验收 什么叫「做完了」？
实现 现在代码差在哪？

产品侧关心用户价值、边界场景；代码侧关心模块、接口、状态机。两者之间天然有缝。更麻烦的是，这条缝往往藏在 senior 工程师脑子里、会议记录里、某条 Issue 评论里——AI 读不到。

所以要做的第一件事，不是写更长的 PRD，而是把两类事实分开，别让 AI 混读：

事实类型 权威来源 AI 应如何使用
目标 / 产品意图 PRD、需求文档 理解「为什么要做」「用户看到什么」
已实现行为 代码 + 差距文档 理解「现在实际怎样」
接口 / 领域形状 types、OpenAPI、Proto 理解「数据结构长什么样」
完成判定 测试、门禁命令 理解「算不算做完」

常见误区有两个方向：把 PRD 规划写成「已实现」；或让 AI 只读代码却按 PRD 改产品行为。差距文档（如 IMPLEMENTATION_STATUS.md）要显式写清：

AI 读到这段，就知道：接下来的改动是在补差距，还是在改产品定义——这是两种完全不同的任务。

3. DSL 的角色：桥梁，不是又一门语言

很多人听到 DSL，会想到规则引擎、代码生成器、一门新语法。在这个语境下，更准确的说法是：

DSL 是中间层：比 PRD 精确（同一概念固定 id，隐含规则写出来），比代码 接近业务（用领域词，不是框架细节），且 可校验（能跑 linter、和代码枚举对比）。

广义上，OpenAPI、JSON Schema、状态机 YAML、验收配置都是 DSL。关键是：领域词汇 + 结构化 + 可校验。

它与术语表、契约、差距文档互补：

手段 和 DSL 的关系
术语映射表 DSL 的关键字/枚举就是术语的正式名
契约层 DSL 生成或约束 OpenAPI / Schema
状态机表 常见形态：pending → paid → cancelled
差距文档 DSL 描述「应该怎样」；差距 = DSL 与代码的 diff
完成判定 acceptance DSL 驱动 verify 命令

什么时候值得上？ 状态机、审批流、定价规则、多 Agent 长期协作的域。什么时候不必强行上？ 一次性 UI 文案、探索期需求、规则少且变更快——维护成本高于收益。

原则只有一条：只有当你愿意把 DSL 当「权威定义」并维护同步时，它才值得。

4. 推荐形态：小 DSL 组合，别造「伪代码 PRD」

5. 完整 DSL 的七层：不是一门语法，是一套体系

「完整 DSL」在仓库里是一个体系（DSL System），可分成七层：

最小完整的判定标准（缺一则不算「完整 DSL」）：

  1. 有 Schema 约束语法
  2. 有稳定 id 与产品 label
  3. 有业务语义（不只是名词表）
  4. 有 status/gap
  5. 有到代码或测试的 mapping
  6. 有自动化校验（至少 CI 一条命令）

没有 status: partial 和 gap，AI 会把 DSL 误读成「已全部实现」。没有 mapping，改 t_cancel_paid 时只能全库搜索。

6. 案例闭环：六文件钉住一条需求

以「paid 订单 30 分钟内取消」为例，六文件形成一条链：

flowchart TD
    PRD["PRD 叙述"]
    GLO["glossary.yaml"]
    LC["lifecycle.yaml"]
    RL["rules.yaml"]
    CAP["capabilities.yaml"]
    AC["acceptance.yaml"]
    IMP["IMPLEMENTATION_STATUS.md"]
    CODE["代码 / 测试 / UI"]

    PRD --> GLO
    GLO --> LC
    GLO --> RL
    GLO --> CAP
    LC --> AC
    RL --> AC
    CAP --> AC
    AC --> IMP
    IMP --> CODE

7. 与代码的三种集成深度

级别 做法 适合
L1 文档型 DSL 只做人/AI 读，人工对照代码 小团队、探索期
L2 校验型 CI 跑 domain:lint，对比代码枚举/路由 推荐起点
L3 生成型 从 lifecycle 生成 types、测试骨架 规则稳定、重复多的域

9. 落地最小文件结构

落地步骤：

  1. 从模板复制到项目 domain/{feature}/
  2. 将示例路径替换为真实代码路径
  3. 增加 domain-lint：校验 id 互引、status 与测试是否一致
  4. 在 AGENTS.md 写入文档索引与冲突优先级
  5. 新需求优先改 DSL，再改代码——而非反过来

10. 常见坑

  1. DSL 语法过重——从小组合起步，不要一上来造语言
  2. 和 PRD 双写——DSL 应是提炼，不是复制粘贴
  3. 没有 status/gap——AI 会把规划当实现
  4. 不可校验——纯 Markdown 表会漂移
  5. 一个 DSL 管 UI + 后端 + 运维——按职责拆分

总结

要点 一句话
问题 产品语言与代码之间有翻译损耗，AI 没参加过需求会
方法 分层事实 + 术语映射 + 小 DSL 组合 + 差距文档
形态 glossary / lifecycle / rules / capabilities / acceptance
集成 从 L2 校验型起步，CI 跑 domain:lint
治理 冲突优先级写进 AGENTS.md；任务带锚点 id

主题本质不是再造一门语言，而是用同一组 id 把产品故事、业务规则、权限、测试和实现差距钉在同一张网上——让 AI 知道改哪、改到什么算完、哪里还不能假装做完。

行动建议

  1. 明天就能做：选一个你正在和 AI 协作的域（订单、权限、计费均可），建 glossary.yaml + 一份 acceptance.yaml，把现有差距写进 IMPLEMENTATION_STATUS.md。
  2. 本周可做：给核心 transition 补上 mapping.code 和 mapping.tests，在 AGENTS.md 写明冲突优先级。
  3. 值得投入：跑 L2 校验——domain:lint 对比 DSL 枚举与代码枚举，比把 prompt 写长一百倍更有效。

工具会变，把隐性知识写成结构化上下文的能力不会过时。你知道该让 AI 听谁的、改哪、怎样算做完——这才是 AI 时代工程负责人该补的一课。
