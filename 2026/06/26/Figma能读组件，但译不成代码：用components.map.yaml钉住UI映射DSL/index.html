Figma 能读组件，但译不成代码：用 components.map.yaml 钉住 UI 映射 DSL

1. 先讲结论

团队已经在 Figma 里维护了和代码对齐的组件库。你把订单列表页的设计稿 URL 丢给 Cursor，Figma MCP 能老老实实告诉你：这里有一个 Button/Primary，Disabled=true，Label="取消订单"。

然后呢？

AI 仍然要猜：import 路径是 @acme/ui/Button 还是 @/components/Button？Figma 的 Variant=Primary 对应代码里的 variant="primary" 还是 type="primary"？文案是塞 children 还是单独的 label prop？

这不是模型笨，是信息本来就不在设计稿里。 MCP 读得到设计语义，读不到你们仓库的 import 约定。我的结论是：在 Figma 和页面代码之间，维护一份 **components.map.yaml**——把组件集名、Property、Variable 翻译成本地组件和 props。这份文件就是 UI 侧的 DSL；MCP 负责读，map 负责译，行为 DSL 负责「什么时候能点」。

姊妹篇《当 AI 读不懂你的 PRD》讲的是行为链（lifecycle、rules、capabilities）。本文讲表现链——设计稿怎么稳定落成代码，而不靠 AI 每次重新猜一遍。

2. 为什么 MCP 不够，还要一份 map

Figma MCP 拉 URL 时，通常能拿到：

MCP 能提供 仍需要对齐文件
组件实例名 Button/Primary 代码 import 路径 @acme/ui/Button
Property Variant=Primary 代码 variant="primary"（大小写、命名）
Property Label 文本 代码 children 还是 label prop
Variable action/primary 代码 tokens.color.action.primary
页面级组合（按钮 + 业务逻辑） wrapper 组件路径

components.map.yaml 的作用 = 翻译层。它不是重复 Figma，也不是替代 MCP。

常见误区：以为「组件库已经对齐了，MCP 够用了」。对齐的是设计侧语义；仓库侧约定（路径、prop 名、token 引用）仍然要有一份显式映射，否则 AI 每次还原都会漂移。

3. 与 PRD / 行为 DSL 的分工

问题 谁回答
取消按钮 Primary 样式 Figma + MCP + components.map#button
variant / disabled props 怎么写 components.map.yaml
paid 超 30 分钟按钮 disabled capabilities.yaml guards（行为 DSL）
点击后调什么 API lifecycle.yaml（行为 DSL）
间距 token 16 vs 20 tokens.map.yaml + UI_IMPLEMENTATION_STATUS

UI DSL 不管业务规则；行为 DSL 不管 import 路径。

交汇点：components.map#order_cancel_button 的 links.capability 指向 capabilities#cap_buyer_cancel——页面上「长什么样」和「什么时候能点」在此挂钩，但各自维护。

4. 体系总览：三文件 + MCP

和姊妹篇的六文件行为链并列，UI 侧最小三件套：

文件 回答的问题
components.map.yaml Figma 组件 → import 谁、props 怎么填
tokens.map.yaml Figma Variable → 代码 token
UI_IMPLEMENTATION_STATUS.md 稿 vs 代码 UI 还差什么

5. 第一层：components.map.yaml（核心 UI DSL）

职责：每条 Figma 组件集 ↔ 本地组件 ↔ props 映射。

5.1 结构说明

块 作用
meta.figma_library 组件库 file_key，MCP 定位用
components[].figma.component_set 与 Figma 组件集名一致
components[].figma.node_id 可选，库大时稳定锚点
components[].code.import AI 应使用的 import
components[].code.path 仓库内路径
components[].props[] Figma Property → 代码 prop
status / gap 映射或实现是否同步

5.2 基础组件：1:1 映射

MCP 返回 Button/Primary, Disabled=true, Label="取消订单" 时，查 #button 即可生成基础代码。

5.3 页面级 wrapper：业务按钮

要点：

  • 基础组件映射 1:1
  • 业务按钮用 wrapper 条目 + links.capability 接行为 DSL
  • Figma 稿上的 disabled 是视觉态；真实 disabled 由 capabilities guards 驱动

稿上按钮可点、规则说应 disabled 时——以 capabilities 为准，UI 实现 disabled 态。这和姊妹篇里的「实现漂移」是同一类问题，只是发生在 UI 层。

6. 第二层：tokens.map.yaml（可选但推荐）

职责：Figma Variables 与代码 Design Tokens 对齐。

命名已完全一致时，文件可以很薄，只记 status/gap；命名不一致时必须显式映射：

MCP 读到样式值时，AI 查表决定用 token 还是硬编码——应优先 token。差距记在 UI_IMPLEMENTATION_STATUS，别藏在某个 PR 评论里。

7. 第三层：UI_IMPLEMENTATION_STATUS.md

职责：汇总 Figma 稿 vs 代码 UI 的差距，不重复行为规则全文。

与 IMPLEMENTATION_STATUS.md 并列：

文档 记录
IMPLEMENTATION_STATUS 规则、API、状态机
UI_IMPLEMENTATION_STATUS 还原度、组件映射、token 漂移

条目应链接 components.map#id 与设计 node-id，方便 AI 和人都沿链定位。

给 AI 的任务模板可以长这样：

产品意图 + 设计锚点 + 行为锚点 + 差距——和姊妹篇行为 DSL 的任务描述是同一套思路，只是入口从 glossary#order_cancel 换成了 components.map#order_cancel_button。

9. 与 Code Connect 的关系

方式 定位
components.map.yaml 轻量、人/AI 可读、全团队可维护
Code Connect (.figma.ts) 映射写入代码仓，MCP 可直接返回 snippet

二者是同一 DSL 的不同载体，不是二选一的对立方案：

  • 起步：YAML 映射表 + MCP，成本低、改起来快
  • 规模化：关键组件补 Code Connect，MCP 直接吐 snippet
  • CI：domain:ui-check 校验 Figma 组件集名 ⊆ map 条目

我见过团队一上来就铺 Code Connect，维护成本反而拖慢迭代；也见过只有 MCP、没有 map，每次还原 import 路径都不一样。先 YAML，再按需上 Connect，通常更稳。

10. 维护时机：跟着组件库走，不跟着每个页面走

事件 更新什么
Figma 新增组件 / variant components.map.yaml + 代码
代码改 props 名 components.map.yaml
Figma Variable 改名 tokens.map.yaml
只改页面布局、仍用库组件 通常不改 map
业务规则变（何时 disabled） capabilities.yaml，非 map

UI DSL 跟着组件库版本走。别把每个页面的布局微调都写进 map——那是 MCP 读实例该干的事。

11. 冲突优先级（UI 相关）

写入 AGENTS.md，和姊妹篇行为链合并成完整优先级：

行为与表现冲突时：capabilities 赢；map 和 MCP 负责把 disabled 态画对，不是决定能不能点。

12. 落地最小结构

落地步骤：

  1. 从模板复制 templates/ui/ 到项目 domain/ui/
  2. 把示例路径换成真实 import 和 wrapper 路径
  3. 先钉 3–5 个高频组件（Button、Input、Modal…），别一次铺全库
  4. 在 AGENTS.md 写入 UI 冲突优先级与文档索引
  5. 新页面还原任务：先查 map，再写代码——而非让 AI 裸读 Figma 硬猜

总结

要点 一句话
问题 MCP 读得到设计实例，猜不出 import 和 props 约定
方法 components.map.yaml 作翻译层；tokens.map.yaml 对齐 Variable
分工 UI DSL 管表现；行为 DSL 管何时何能点；links.capability 挂钩
差距 UI_IMPLEMENTATION_STATUS.md 与行为差距文档并列
演进 YAML 起步 → 关键组件补 Code Connect → CI 校验

主题本质：对齐的 Figma 组件库解决「设计侧语义」；components.map.yaml 解决「翻译到本地代码」——这就是 UI 侧真正需要维护的那一份 DSL。

行动建议

  1. 明天就能做：选一个你正在用 MCP 还原的页面，建 components.map.yaml，把页面上 3 个组件的 import 和 props 写进去。
  2. 本周可做：给业务按钮加 wrapper 条目 + links.capability，在 UI_IMPLEMENTATION_STATUS.md 记一笔当前还原差距。
  3. 值得投入：和姊妹篇行为 DSL 共用同一套 status/gap 元数据——AI 才知道哪里能假装做完、哪里还不能。

设计稿会变，组件库会迭代，但 「Figma 实例 → 本地 import」这张网 一旦钉住，MCP 才真正从「能看」变成「能译」。
