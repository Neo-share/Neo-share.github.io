写给谁：实践 AI 主力研发的开发者、技术负责人
一句话：把模糊目标变成 AI 可执行任务，让人只在门禁签字。
核心命题：阶段 + 门禁 + 状态机 + 并行切片，是 AI 时代交付稳定的 How。

姊妹篇 《AI 时代的软件开发最佳实践》 讲了 Why/What。本文是 How——一条需求从进池到上线，经过哪些阶段、哪些节点必须人确认、哪些可交给 AI。

工程落地（Monorepo 任务 YAML、Scope、CI 门禁）见 《AI Agent 自动化工程实践》。

1. 编排师在流程里干什么？

AI 研发交付编排师编排目标、约束、AI 任务和门禁，不替代业务/技术/发布 owner 的最终决策。

接管：需求梳理、方案草稿、拆任务、指挥 AI 实现/测试/修 BUG、准备门禁材料。
不替代：立项、架构取舍、发版回滚、业务验收。

2. 流程总览：五个阶段、十四个节点

flowchart LR
  Start([开始]) --> Demand["需求与立项"]
  Demand --> Design["方案与设计"]
  Design --> Plan["拆分与排期"]
  Plan --> Build["开发与验证"]
  Build --> Release["发布与监控"]
  Release --> Improve["验收与沉淀"]
  Improve -->|"新需求"| Demand
阶段 名称
1–3 需求收集、立项排期、需求评审
4–7 技术调研、方案评审、契约评审（条件）、UI 评审（可选）
8 任务拆分与估时
9–11 代码实现、自测与评审、自动化测试与联调
12 BUG 闭环（贯穿，非单点）
13–14 发布审批与灰度、线上监控与复盘

3. 核心、条件核心与可选

  • 核心：1–5、8–11、13–14 默认必经
  • 条件核心：跨模块/契约变更 → 阶段 6 契约评审
  • 可选：UI、PoC、安全/性能/迁移、合规、运营准备、事故复盘
  • 贯穿：BUG 修复闭环

4. 高频循环

  • 需求循环：范围/价值/排期问题 → 回到立项
  • 方案循环：不可行/契约风险 → 回到需求或方案
  • 验证循环：自测/Review/测试/联调失败 → 回到开发
  • BUG：待确认 → 待修复 → 修复中 → 待回归 → 已修复

5. 并行：按切片，不按岗位

需求评审后可并行：调研、PoC、UI 梳理、测试策略、契约草案。

任务拆分后按纵向切片并行：主流程、权限边界、列表查询、异常态——共享基础先定最小契约。

契约稳定后可提前写自动化测试；发布说明、监控、灰度、回滚可在开发中后段并行准备。

6. 推荐工作项状态机

验证子状态：自测 → Review → 自动化测试 → 环境验收（失败回开发）。
发布子状态：CI → 部署 → 灰度 → 正式发布（异常回开发或回滚决策）。

7. 七道阶段门禁

需求、方案、UI（触发时）、开发、测试、发布、验收——每道门禁有明确准出标准（价值/范围/契约/CI/监控等）。

8. 九个人工确认节点

立项、需求、方案、契约、UI（触发时）、合并、发布、放量回滚、业务验收——必须人工确认，不可默认自动化。

9. 自动化边界

类型 示例
AI 可自动 摘要、草稿、测试用例、CI 分析、明确 BUG 修复
必须人工 立项、方案取舍、合并、发布、验收
禁止默认自动 改生产数据、发版回滚、改权限密钥、对外正式通知

10. 与工程实践的衔接

流程 工程落地
任务拆分 agents/*.tasks.yaml
范围控制 Scope 校验
开发门禁 CI quality gate
并行 Git Worktree 多 Agent

总结

  1. 14 阶段 + 五段主链路
  2. 七道门禁 + 九个确认节点
  3. 切片并行 + 贯穿 BUG 闭环
  4. 三张自动化边界清单

行动建议

  1. 画团队主状态机，标缺失准出标准
  2. 列门禁表：九节点谁签字、AI 备什么材料
  3. 选一个需求试纵向切片并行
