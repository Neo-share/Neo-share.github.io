先讲结论：这不是可选潮流

如果你过去十年做的是 Web 应用，你大概率熟悉这条主线：

表单 + 列表 + 详情页 → REST/GraphQL → 状态管理 → 部署上线

这条链路不会一夜消失，但产品交付的默认假设正在变：用户不再只想「点哪里进哪里」，而是想用自然语言完成目标，由系统理解意图、编排步骤、调用能力、给出结果。

我认为：几乎所有现有 Web / 移动 / 桌面应用，都会在 3–5 年内经历「Agent 化」改造——不是全部变成聊天机器人，而是把「理解意图 + 自主执行 + 可审计」嵌进核心流程。

对 Web 开发者而言，这意味着：

过去你优化的 未来你要优化的
首屏、路由、组件复用 意图识别准确率、工具调用成功率
表单校验与字段联动 任务分解、步骤可回滚、人机确认点
API 延迟与缓存 Agent 上下文、Token 成本、流式响应体验
权限按钮显隐 工具级授权、操作审计、越权拦截

Agent 应用开发，就是 Web 开发的下一站。 越早理解演进路径，越不会在「给老系统贴一个 Copilot 侧边栏」时走弯路。

什么是「Agent 应用」？

一句话定义

Agent 应用：以大型语言模型（LLM）为推理核心，能够根据用户目标自主规划、调用工具、读写状态、在多步执行中保持上下文的软件系统。

它和「带 AI 功能的传统应用」的差别，不在于有没有模型，而在于谁主导流程。

flowchart TB
  subgraph traditional["传统 Web 应用"]
    U1["用户"] --> UI1["固定 UI 流程"]
    UI1 --> API1["确定性 API"]
    API1 --> DB1["数据库"]
  end

  subgraph agent["Agent 应用"]
    U2["用户（自然语言目标）"] --> ORCH["编排层 / Agent Runtime"]
    ORCH --> LLM["LLM 推理与规划"]
    LLM --> TOOLS["工具：API · DB · 搜索 · 代码 · 浏览器"]
    TOOLS --> STATE["会话状态 · 任务状态 · 审计日志"]
    ORCH --> UI2["动态 UI / 流式反馈 / 确认卡片"]
  end

四个关键特征（Web 开发者版）

  1. 意图驱动：用户说「帮我把上周未发货的订单批量提醒客服」，而不是点开 5 个筛选器。
  2. 工具调用（Tool Use）：Agent 不「猜」数据，而是通过你暴露的 HTTP、tRPC、SQL、搜索等受控接口拿真数据、做真操作。
  3. 多步执行：一次对话可能触发查库 → 生成草稿 → 等人确认 → 批量写入；需要任务状态机，不是单次 request/response。
  4. 可治理：每次调用谁、改了什么、为何失败，必须可追溯——这和 Web 里的「操作日志 + 权限」是一脉相承的。

「应用 Agent 化」发生在哪几个层次？

不必一夜之间重写产品。业界常见的演进是由浅入深三层：

flowchart LR
  L1["L1 体验层\nCopilot / 侧边栏 / 命令面板"]
  L2["L2 能力层\nAPI 工具化 · MCP · Function Calling"]
  L3["L3 架构层\n意图路由 · 任务编排 · 人机协同"]

  L1 --> L2 --> L3
层次 用户感知 你要改什么 典型工期
L1 体验层 「有个 AI 助手帮我查/写」 前端加对话 UI；后端加 /chat 代理 天～周
L2 能力层 「AI 能直接办业务」 把现有 REST 封装成 Tool Schema；统一鉴权与幂等 周～月
L3 架构层 「打开应用就是完成任务」 任务编排、状态机、审批流、审计与回滚 月～季

误区：长期停在 L1——所有业务仍靠人手点页面，AI 只能「回答问题」。
正路：L1 获客、L2 沉淀可调用能力、L3 让核心路径走 Agent 编排。

交互范式：从「页面导航」到「目标完成」

优点是确定性强、可测试、性能好；缺点是学习成本高——新用户要记住菜单在哪、字段什么意思。

Agent 化 Web：界面即「执行结果的投影」

用户给出目标，系统生成当前最优的 UI（可能是表格、卡片、确认框、流式日志）：

对前端的影响：

  • 组件库仍然重要，但更多「按需组装」，而不是固定 wizard。
  • 流式 UI（SSE / WebSocket）成为标配：用户要看到 Agent「正在查库 / 正在写文件」。
  • 确认点（Human-in-the-loop） 是一等公民：涉及钱、权限、删除时，必须卡住等人点确认——这和表单里的「二次确认」同源，只是触发条件由 Agent 决策。

三种 UI 模式并存（未来 3 年的常态）

模式 适用场景 Web 实现要点
对话式 探索、咨询、模糊需求 消息列表 + 工具调用卡片 + Markdown/代码块渲染
命令式 高频专家用户 Cmd+K 面板、斜杠命令、自然语言 → 结构化动作
传统页面 强合规、精确录入 保留表单；Agent 负责预填、校验解释、批量生成

好的 Agent 化产品不是「去掉页面」，而是让页面与对话共享同一套业务能力。

技术栈变化：Web 开发者会碰什么？

前端

领域 传统重点 Agent 化后新增
状态 Redux / Zustand 页面状态 会话状态、任务 ID、工具调用历史
网络 fetch / React Query SSE 流、分片渲染、中断与重试
组件 表单、表格 Tool Result 渲染器（表格/图表/ diff / 文件预览）
安全 XSS、CSRF 提示注入防护、敏感操作二次确认 UI

技术选型上，React / Vue / Next / Nuxt 都仍适用；变化的是边界：前端不再只调 BFF，还要展示「Agent 正在调用哪些工具、结果是否可信」。

实践要点：

  • Schema 先行：Zod / JSON Schema 同时服务 LLM、TypeScript 与文档。
  • 幂等与限流：Agent 可能重复调用同一工具。
  • 权限下沉到工具级：「能聊天」≠「能删库」。

编排与运行时

组件 作用
Agent Runtime 管理 prompt、模型路由、工具循环、超时
任务队列 长任务异步化（生成报告、批量导入）
状态存储 会话、计划步骤、checkpoint（便于断点续跑）
可观测性 每次 tool call 的 input/output/latency/cost

开源与商业方案都在快速成熟（LangGraph、Temporal、各类 Agent SDK、Cursor 等）。Web 团队不必自研模型，但必须自研领域工具与治理策略。

与 MCP、OpenAPI 的关系

  • OpenAPI / tRPC：描述「你的业务 API 长什么样」。
  • MCP（Model Context Protocol）：描述「外部能力如何以标准方式接到 Agent」。

Web 团队的最小行动：先把内部高频 API 整理成 Tool 清单（名称、描述、参数、副作用、权限），再考虑是否挂 MCP Server。否则 Agent 只能在文档里「幻觉」业务。

现有应用如何渐进 Agent 化？

下面是一条可落地的迁移路径（适合已有用户量的 Web 产品）：

flowchart TD
  A["盘点高频用户目标\n（不是页面，是任务）"] --> B["选 3–5 个只读场景\n做 L1+L2 PoC"]
  B --> C["统一 Tool 层 + 鉴权 + 审计日志"]
  C --> D["引入任务状态机\n支持多步与失败恢复"]
  D --> E["核心写操作加审批流\nHuman-in-the-loop"]
  E --> F["评估成本与 SLA\n模型路由 / 缓存 / 小模型兜底"]

第一步：用「用户任务」倒推，不是用「页面」正推

写下用户真正想完成的句子，例如：

  • 「把这周转化率下降的 SKU 列出来并解释可能原因」
  • 「根据模板生成活动页并预览」
  • 「把测试环境数据库同步到 staging（危险）」

每个任务映射到：需要哪些读接口、哪些写接口、哪些必须人工确认。

第二步：Tool 化，而不是 Prompt 化

不要把业务逻辑写进 prompt 里「你帮我算一下折扣……」。
折扣规则应在代码里；Agent 只负责选择调用 calculate_discount 并展示结果。

这样：

  • 单元测试仍然覆盖业务代码；
  • Agent 行为可通过「工具是否被正确调用」来评估；
  • 换模型时业务风险可控。

第三步：人机协同与审计

Web 应用已有登录态、RBAC、操作日志——Agent 化后应升级为：

机制 说明
工具级 RBAC 角色 A 只能 read_*，角色 B 才可 write_*
审批流 写操作生成 pending task，负责人在 UI 批准
证据链 用户问题 → 计划 → 每次 tool 输入输出 → 最终结果
回滚 批量写操作支持 dry-run 与撤销

第四步：体验与成本工程

  • 流式反馈：减少「黑盒等待」带来的不信任。
  • 模型路由：简单分类用小模型，复杂规划用大模型。
  • 缓存：相同只读查询命中 Redis，避免重复 Token。
  • 失败降级：工具超时 → 明确告知用户并给出「改用手动页面」的链接。

Web 开发者现在该补哪些能力？

短期（1–2 个月可上手）

  1. Tool / Function Calling 设计：会写清晰 description、严格 parameters、合理错误码。
  2. 流式接口与前端渲染：SSE、分块 JSON、中断请求。
  3. Prompt 与产品边界：哪些交给模型、哪些必须代码兜底。
  4. 基础评估：同一用户目标，多轮测试「是否调对工具、是否越权」。

中期（3–6 个月形成团队能力）

  1. 任务编排与状态机：多步任务、失败重试、checkpoint。
  2. 安全：提示注入、敏感数据脱敏、工具沙箱。
  3. 可观测性：trace 每次对话的成本、延迟、工具失败率。
  4. 评测体系：黄金数据集 + 自动化回归（发布前跑一轮 Agent 场景）。

长期（差异化竞争力）

  1. 领域 Agent 架构：电商、CRM、DevOps 各自的最优工具粒度。
  2. 与 CI/CD、工单、监控打通：Agent 不只是 UI，而是运维与研发闭环的一部分。
  3. 多 Agent 协作：规划 Agent、执行 Agent、审查 Agent 分工（对 Monorepo 团队尤其相关）。

风险与边界：理性 Agent 化

风险 表现 Web 侧应对
幻觉 编造不存在的数据 强制 Tool 取数；UI 标注「未验证」
越权 Agent 调了不该调的接口 工具级鉴权 + scope 校验
成本 Token 账单失控 配额、缓存、模型分级
延迟 多轮工具调用慢 异步任务 + 进度页
合规 日志含 PII 脱敏、 retention、区域部署

不是所有功能都适合 Agent 化。 强监管表单录入、毫秒级交易、严格确定性计算，仍应以传统代码为主，Agent 做辅助。

若你已在 Monorepo 或复杂全栈项目里让 AI Agent 参与改代码、跑 CI、控范围，可继续阅读同系列的工程化实践文：

两篇合在一起，对应 Agent 时代的两个战场：

flowchart LR
  P["产品运行时\n用户目标 → Agent 执行"] 
  E["研发运行时\n需求 → Agent 改代码 → CI"]

  P --- E

产品 Agent 化 + 研发 Agent 化，会同时发生在一家互联网公司里。Web 开发者往往两边都会碰到。

总结：给 Web 开发者的一句话

未来不属于「会做页面」或「会做 API」的单点技能，而属于能把业务能力封装成 Agent 可调用的、可治理的、可观测的工具，并在体验层做好人机协同的人。

应用都会 Agent 化，但不是把应用变成聊天软件，而是：

  1. 用户侧：从导航菜单走向目标完成；
  2. 架构侧：从 CRUD 接口走向 Tool + 编排 + 审计；
  3. 团队侧：从只交付功能走向交付「可被 AI 安全调用」的能力。

如果你今天维护的是一个正常的 Web 应用，现在就可以从 L1 的体验试点和 L2 的 Tool 清单开始——等竞争对手把核心流程做成「一句话办完」，再补功课的成本会高一个数量级。

你可能已经烦过这些

网购久了，大概都遇到过类似时刻：

  • 想退货，却在「我的订单 → 申请售后 → 选原因 → 上传凭证」里绕了好几圈；
  • 想买一件「适合送长辈、预算 300 左右、别太贵但要有面子」的礼物，搜了半小时，收藏夹里堆了十几款，还是拿不定主意；
  • 大促凑满减，计算器按了 N 遍，最后发现有一单忘了领券；
  • 物流显示「已签收」，你明明没收到，客服排队 20 分钟，还要重复说一遍订单号。

这些事的共同点：App 功能很多，但你要自己当「导航员」——知道入口在哪、规则是什么、下一步点哪个按钮。

行业里最近常聊一个词：Agent 化。听起来像技术黑话，其实可以翻译成一句更直白的话：

以后你打开 App，更像在跟一个能办事的助手说话，而不只是在翻菜单。

下面全部用网购举例，尽量不涉及代码和架构。

先用人话解释：什么是「Agent 化」？

你可以把 Agent 理解成：不只回答问题的助手，而是能替你查、替你办、办到一半还会来问你「这一步要不要继续」的那种。

对比项 今天的常见体验 Agent 化之后（理想状态）
你怎么用 自己找入口、填表单、对规则 说目标，例如「帮我把上周那单退了」
App 怎么回应 固定页面：列表、按钮、弹窗 先理解意图，再按步骤去查订单、填申请、等你确认
出错时 你自己猜是哪一步错了 助手说明卡在哪，给出可选方案

注意：不是把 App 变成一个聊天框就完事。 该让你签字确认的地方（比如涉及退款金额、地址修改）仍然会停一下；该看商品图、看评价的时候，页面也还在——变的是谁负责串联这些步骤。

电商为什么可能是「第一批」让你感受到的变化？

网购有几个特点，和「助手替你跑腿」特别合拍：

  1. 步骤多：选品、领券、下单、支付、物流、售后——一条链路上环节长。
  2. 规则杂：满减、会员价、地区限购、退换条件，普通人很难一次记全。
  3. 重复劳动多：查物流、改地址、催发货、问「能不能价保」，本质都是「带着订单信息去问系统」。
  4. 你其实早就想要「一句话办完」：「跟客服说人话」的需求，比很多 App 都强烈。

所以当我们说「未来应用都会 Agent 化」，电商 App、外卖、出行订票这类「强流程」产品，往往比纯娱乐类 App 更早走出一步——不是因为它更酷，而是因为用户痛点本来就在「流程太长」。

五个日常场景：今天 vs 明天

场景 A：送礼不知道买啥

今天：打开搜索框，换关键词「母亲节礼物」「实用 礼盒」，刷评价、比价格、问朋友，最后下单时还可能买错尺码或颜色。

Agent 化之后（设想）：

「想给 60 岁妈妈买生日礼物，她腰不好、不爱花哨，预算 500 以内，要能在三天内送到杭州。」

助手可能会：根据历史偏好（若你授权）排除不合适的品类 → 筛出几款候选 → 用卡片展示对比（价格、评价摘要、是否支持次日达）→ 你点选一款 → 它帮你检查地址和优惠券 → 下单前弹确认页，你点「确认支付」才真正扣款。

你省掉的是反复搜索和比对；你保留的是最后拍板的权力。

场景 B：退货 / 换货

今天：找到订单 → 判断是否在七天无理由内 → 选退货原因 → 拍照上传 → 等审核 → 自己填快递单号。

Agent 化之后（设想）：

「这双鞋磨脚，想退掉，上周三收到的。」

助手：定位订单 → 说明是否符合退换规则 → 若需要，引导拍两张照片 → 生成退货申请草稿 → 你确认后提交 → 若有上门取件，直接帮你预约时段。

若规则不允许退（比如已拆封的贴身衣物），它会先说清楚原因和替代方案（换尺码、部分补偿券等），而不是让你填完表才被驳回。

场景 C：物流异常

今天：复制快递单号 → 打开物流页 → 发现卡住 → 进客服 → 重复报订单号、地址、问题描述。

Agent 化之后（设想）：

「显示签收了但我没收到。」

助手：拉取该订单物流轨迹 → 判断是「驿站代收」「投递错误」还是「时间差」→ 若需要，代你发起工单或联系承运方，并把进度用你能看懂的话同步回来。

场景 D：大促凑单

今天：购物车算满减、跨店凑单、会员叠加，经常「差 23 元」——然后为了凑单多买了一样不需要的东西。

Agent 化之后（设想）：

「我想凑到 300 减 40，别让我买垃圾凑数。」

助手：读取购物车和可用券 → 推荐真正可能用得上的小件（纸巾、常购耗材等）或提示「分开下单更划算」→ 给出一页对比：现在下单 vs 放弃满减，各花多少钱。

重点不是「AI 帮你买更多」，而是把算账这件事接过去。

场景 E：价保 / 发票 / 改地址

这类需求特点：规则清晰、操作重复、但入口深。正是 Agent 适合接手的「跑腿型」任务——你说目标，它在后台调用原本就有的能力（查价保资格、申请发票、修改未发货订单地址），只在关键节点让你确认。

和「现在 App 里那个 AI 客服」有什么不同？

很多平台已经加了「智能客服」，你大概体验过：

  • 问东答西，三句之后转人工；
  • 只会发帮助文档链接；
  • 一涉及改订单、退钱，立刻说「请您自行操作」。

那种更像问答机器人。Agent 化想解决的是再往前一步：

智能客服（常见） Agent 化（方向）
主要「说话」 在授权范围内真的去查、去办
单次问答 多步任务：查 → 比 → 申请 → 确认
失败了让你自己找入口 说明卡在哪，并给出下一步可选项

当然，今天市面上绝大多数 App 还没到后面这一列。上面写的很多是「正在发生和即将发生」的方向，不是承诺每一款明天就能用。

你需要知道的：好处与顾虑

可能带来的好处

  • 少记规则：满减、退换条件、会员权益，由助手按场景解释。
  • 少点几次：高频琐事（查物流、催发货、开发票）一句话发起。
  • 决策更轻松：对比、摘要、风险提示，比你自己翻几十条评价省时间。

合理的顾虑（也值得产品认真做）

  1. 会不会乱下单、乱退款？
    正规设计会在支付、退款、改地址等节点强制你确认；不应「一句话就悄悄扣款」。

  2. 隐私怎么办？
    助手若要「猜你喜欢」，需要订单、地址、浏览记录——应明确告诉你用了哪些数据、能否关闭，而不是默认全开。

  3. 搞错了谁负责？
    比「聊天说错了」更严重的是「办错了」。所以可追溯、可撤销、能找人工兜底，和今天电商的售后体系一样重要，甚至更重要。

  4. 会不会反而更难用？
    若只加一个聊天气泡，却解决不了问题，就是添乱。好的 Agent 化是该说话的时候说话，该出页面的时候出页面——不是把一切塞进对话框。

App 界面会消失吗？

大概率：不会。

商品图、尺码表、评价、支付密码、物流地图——这些仍然适合用「页面」呈现。变的是：

  • 你不必从首页开始自己找路；
  • 助手负责在后台串联查库存、领券、填地址；
  • 界面变成「当前这一步你需要看的东西」，而不是固定菜单树。

可以粗浅理解成：

店员再熟，付钱仍在你。

这和「未来所有 App 都会 Agent 化」有什么关系？

电商只是例子。同样的逻辑会出现在：

  • 外卖：「两家店一起点，总配送费怎么最省」
  • 出行：「周五晚七点到虹桥，要便宜但也要准时，帮我比一下高铁和网约车」
  • 银行 / 理财 App：「这个月多还一点房贷会不会更划算」——前提是严格确认与合规

共同趋势是：从「人适应 App 的菜单结构」，转向「App 适应人的目标」。
电商因为离钱近、流程长、用户基数大，往往是最先被讨论、也最先试水的领域之一。

你现在可以怎么想这件事？

不必追每一个新名词。作为普通用户，记住三句就够：

  1. Agent 化 ≈ App 从「等你点」变成「能替你跑流程，但重要步骤仍要你点头」。
  2. 电商很可能是你最早感知到变化的一类 App——因为退货、物流、凑单太费神了。
  3. 值得警惕的不是 AI 太强，而是 AI 乱办还不让你知道——好的产品设计，一定让你看得懂、管得住、退得回。

若你好奇「App 背后工程师在怎么实现这些」，可以读本站面向开发者的姊妹篇：Web 应用的 Agent 化：面向开发者的未来演进——那是同一主题的另一面。

也欢迎你在评论区留一句：你最希望哪个购物 App 先帮你办掉哪一件烦心事？ 退货、凑单、还是找礼物——也许你的答案，就是下一代产品该先做的地方。

小结

问题 简短回答
Agent 化是什么？ App 能听懂目标，并多步帮你办事，而不是只展示页面
为什么用电商举例？ 步骤多、规则杂、重复劳动多，痛点清晰
界面会没了吗？ 不会，但菜单导航的压力会小很多
安全吗？ 取决于产品是否做好确认、权限和售后——这和今天网购一样关键

网购从 PC 时代「搜索 + 下单」，到手机时代「随时随地买」，已经变过一次。下一变，可能是从「自己逛」到「说目标、让 App 跑腿」——上面说的 Agent 化，本质上就是这件事。

我是 Neo，做软件开发，长期在电商与加密货币场景里写 TS/React/Vue/Go，现在更关心 Agent 如何改变应用形态。

一次「小改」牵出一串系统

有一次我以为只是改一个列表页的小交互：多一列筛选、调整一下空状态。结果从接口字段、缓存失效策略，到不同角色的按钮权限，都要跟着动一遍。电商系统里这种「小需求」并不少见——活动、库存、订单、营销规则叠在一起，表面是 UI，底下是一张张牵着的网。

后来在加密货币相关产品里，我又习惯了另一套紧绷感：行情与资产数据要 timely，展示与风控边界要清楚，用户对「数错了」的容忍度极低。两段经历叠在一起，我慢慢形成同一个判断：我关心的从来不只是某个框架怎么用，而是一整套系统怎么稳稳地演进。

而最近一年，Agent 和 Cursor 一类工具把「从意图到改动」的距离拉短了。我既兴奋，也警惕——兴奋的是交付方式在变，警惕的是如果还按「堆功能」的老习惯往上贴 AI，只会多一个不好管的聊天框。这也是我开这个博客、并开始写 Agent 相关文章的直接原因。

从「堆功能」到「能演进」

之前：先把需求做完

早期我的工作方式很典型：产品给 PRD，我拆页面和接口，排期、联调、上线。电商里常见的是大促前赶活动页、库存扣减与订单状态机；币圈相关产品里则是行情展示、资产汇总、充提与风控策略的呈现。目标明确：按时交付、少出线上事故。

卡点：改一处，牵全身

真正磨人的不是写某个组件，而是边界不清。业务规则散落在多个服务里，改一个字段要问三个团队；联调环境不稳定，问题难以复现。需求看起来小，风险却像滚雪球。

现在：先想清楚怎么改、怎么验

现在我更愿意在动手前多问几句：这段逻辑属于谁？谁能改、改完怎么验？能不能分阶段上？对 AI/Agent，我的态度也一样：它是嵌进流程的能力，不是替代思考的捷径。

我宁可慢一点把「谁能改什么、改完怎么验」说清楚，也不追求一次生成看起来能跑的代码。

这句话差不多就是我现在的工作底线。

我是谁：软件开发，主战场在 Web 与后端

  • 角色：软件开发。熟悉电商（活动、交易、商品与订单链路）与加密货币（行情、资产、风控与展示）相关业务和系统特点。
  • 技术栈：TypeScript · React · Vue · Go
    • TS 与 React/Vue：产品界面、复杂交互、BFF 与前端工程化。
    • Go：接口服务、任务与需要稳定吞吐的后端模块。
  • 当前关注：Web 应用的 Agent 化、工程化落地，以及「未来 App 还会长什么样」。

我做事的三条原则

先划边界，再写代码

模块、权限、数据归属先说清，再讨论实现。电商里一个「加按钮」可能涉及运营权限；币圈里一处展示错误可能直接影响用户信任。边界清楚了，Agent 辅助改代码才不会越改越乱。

改造要可渐进

大系统允许分阶段演进：先只读查询、再半自动、最后才谈全自动。Agent 化也一样——不是一夜之间把整站换成聊天界面。我在 《Web 应用的 Agent 化》 里写的「渐进落地」，也是这条原则的外延。

结果要可验证

能预览、能回滚、能审计。尤其是让 Agent 动仓库时，更需要状态机、范围约束和日志——这在 《AI Agent 自动化工程实践》 里有更完整的展开，这里不重复，只强调：能验、能查，比「生成得快」重要。

写给谁：想了解「Neo 到底是干什么的」的朋友、家人
一句话：我是做软件的，主要在网购、电商和数字资产（加密货币）相关的产品里干活——日常不是「神秘地敲键盘」，而是让 App 和网站能稳定、正确地办事。

先用人话：我是干什么的？

如果朋友问「你是程序员吧，具体做啥？」——我会这样答：

我负责把「产品想做的事」，变成手机上、电脑上真正能用的功能，并且尽量别出岔子。

打个比方：

  • 你网购时点的「加入购物车」「领券下单」「查物流」——背后有一整套系统在配合；
  • 你在某个 App 里看余额、转账、查行情——也是同样，界面你看得见，后面还有大量你看不见的衔接工作。

我的工作就在这个衔接里：前面是用户点的按钮，后面是订单、库存、支付、权限、数据对不对——我要保证它们连得上、算得对、改一处不会莫名其妙坏别处。

不是修电脑，也不是单纯「写代码炫技」。更准确的说法是：在复杂业务里，把东西做出来、养得住。

我在哪两个行业里干活？

行业一：电商 / 网购

就是和买东西、卖东西有关的互联网产品。

你可能熟悉这些场景：

  • 大促抢券、凑满减；
  • 下单后看物流、改地址；
  • 退货退款要走好几步；
  • 商家后台要管商品、库存、活动。

这些看起来都是「页面上多一个按钮」，但背后往往涉及：这个按钮谁能点？点了之后扣哪里的库存？订单状态怎么变？退款原路返回还是退到余额？

我在电商领域待得比较久，习惯的是这种环节多、规则杂、一点小改动可能牵一串的环境。

行业二：数字资产 / 加密货币相关

第二个主战场，和比特币、以太坊那一类数字资产有关——常见形态是：交易 App、资产展示、行情页面、充提与风控相关功能。

和电商不同，这里用户特别在意一件事：数字对不对。

  • 余额少显示一位，用户会炸；
  • 行情慢半拍，信任感就掉；
  • 涉及钱的操作，展示、权限、风险提示都要格外清楚。

所以这段经历教会我：不光要「能做出来」，还要「让人放心」。

两个行业听起来离普通人生活都不远——一个几乎人人网购，一个听过名字但不一定自己玩。共同点是：都是强流程、强规则、出错代价高的产品，也正是我日常最熟悉的战场。

我一天到底在忙什么？

朋友最容易误会的是：程序员 = 整天对着黑屏敲神秘符号。

其实更像下面这些事（我按真实比例打个比方，不同项目会有差别）：

大约三成：把需求「做出来」

产品或业务会说：「希望用户能在这里筛选订单」「希望这个页面加载快一点」「希望多一种登录方式」……

我要做的，是把这些话翻译成能上线的功能：页面长什么样、点下去发生什么、失败了提示什么。

大约三成：「改一处，别坏一片」

这是电商、金融类产品最磨人的部分。

举个例子（不用懂技术也能明白）：

运营说：列表上加一列「预计送达时间」。
听起来很小对吧？
但要问：这时间从哪来？所有订单都有吗？老数据怎么办？手机版和网页版要不要一致？有没有权限只能给客服看？

所以我大量时间花在：改之前先想清楚会牵动谁，改之后怎么确认没搞砸。

大约两成：联调、排查、修问题

「我这边下单成功了，你那边库存怎么没减？」「测试环境可以，上线就不行」——这类问题很常见。

有点像几个部门接力赛跑，中间有一棒传丢了，就要顺着整条链路查是谁慢了、谁理解错了。

大约两成：讨论「以后怎么做更稳」

包括：能不能分步骤上线？出错了怎么回退？以后加新功能会不会更难维护？

最近这一年，还多了一块：AI 能不能帮我们把上面这些事做得更快、更稳——而不是变成「看起来聪明、实际上乱改一气」。这也是我写文章、做实验的原因（下面会提到）。

用三个生活比喻，帮你建立画面

比喻一：像大型商场的「后台运维 + 装修 + 接线」

你逛商场，看到的是陈列、收银、导购。
商场背后还有：仓库、会员系统、促销规则、各柜台的结算方式。

我更像在后台做接线、改规则、保证收银和库存对得上的人——不是站在柜台前卖货的，但柜台能不能正常卖，和我有关。

比喻二：像乐队的编曲，不是只弹一个音

一个 App 功能上线，往往要前端（你看到的界面）、后端（服务器里的逻辑）、别的团队提供的接口一起配合。

我只负责其中一个声部，但要听整个乐队——哪个音高了、哪个节奏错了，都要能找出来。

比喻三：像给复杂机器做「可检修的设计」

系统越做大，越不能像一次性搭好的乐高，拼错只能全拆。

我越来越在意：以后还要改很多次，现在能不能留好检修口、说明书、保险丝——这样下次改「加一列筛选」时，不用把整个商场翻一遍。

那「Agent」「AI 改 App」和我有什么关系？

你可能在新闻里听过 AI、大模型、Agent。用完全不技术的话说：

以后 App 可能不只等你点按钮，还能听懂「帮我把那单退了」「帮我挑个送妈妈的礼物」，然后按步骤去查、去办，重要的地方再停下来问你同不同意。

我在电商里泡了很多年，太清楚用户烦的是什么——不是 App 功能少，而是步骤太长、规则太绕，得自己当导航员。

所以我最近的工作重心，除了日常交付，还在想：

  • 这种「能办事的助手」，先用在哪些场景最合适（比如查物流、售后、凑单）；
  • 哪些步骤必须让人亲手确认（涉及付钱、改地址、退款）；
  • 怎么避免变成「聊天挺溜，一办事就乱套」。

若你对「未来购物 App 会怎样变」感兴趣，可以读这篇更具体的：未来的购物 App 会变怎样？——全文用网购举例，没有代码。

我和「纯写代码的人」有什么不同？

诚实说，行业里人也分很多种。我更像这一类：

别人可能更关注 我更常关注
某个工具新不新、酷不酷 业务规则清不清楚
功能能不能 demo 出来 上线后能不能长期改
个人写得多快 团队协作会不会越改越乱
AI 能不能一次生成整站 AI 帮忙时，哪些能自动、哪些必须人把关

这不是说哪种更高明，只是我的战场在复杂业务里——电商、数字资产都是「不能拍脑袋上线」的类型，所以我也会更在意边界、验证、分步走这些事。

若你是同行、想看更偏技术版本的自我介绍，可以看：关于 Neo：从电商与链上到 Agent 化 Web。

这个站 https://Neo-share.github.io 里：

  • 有些文章偏技术，给做软件的人看；
  • 有些像这篇，尽量让非技术朋友也能看懂；
  • 偶尔也写「未来 App 长什么样」这类偏思考、偏趋势的内容。

你不需要懂我用的那些工具名字；若你关心的是「网购会不会更好用」「AI 会不会乱动我的钱」——那正是我会持续写下去的方向。

朋友版 FAQ

问：你是前端还是后端？
答：都可以简单理解成——你看得见的那部分，和看不见的那部分，我都做。 具体怎么分工，看项目需要。

问：你加班多吗？
答：电商大促、线上出问题的时候会更忙；平时看项目和阶段。和很多互联网岗位类似，不是固定 955，也不是每天通宵。

问：你会帮我修电脑 / 装系统吗？
答：可以帮忙看看，但那不是我的主业……主业是做产品和系统。

问：加密货币那行安全吗？是不是在搞诈骗？
答：行业里确实有乱象，所以我更强调合规、风控、展示准确——我做的是正经产品里的软件，不是拉人投资那种。你若不玩，完全没问题；我只借这个例子说明「数字必须对」这类工作要求。

问：AI 会不会取代你？
答：AI 已经在帮我写一部分重复劳动，但需求是不是对的、上线后谁负责、出了问题怎么查——这些仍要人拍板。我把它当更趁手的工具，不是失业预告。
